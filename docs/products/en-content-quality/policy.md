# en版サイト 独自コンテンツ品質確保 方針書

## 問題定義
- **対象ユーザー**: ng-devtools の英語圏訪問者（Google検索経由でツールページ・ガイドページ・トップページにアクセスするエンジニア層）
- **課題**: Search Consoleで `en/`（トップ）、`en/menu`、`en/guide` を含む en 版のほぼ全ページが「クロール済み - インデックス未登録」と判定されている。各ツールヘルプの英語訳が ja 版の機械的な直訳ベースであり、英語圏の検索意図・表現に対して独自性・厚みが不足しているため、Googleから低品質な複製コンテンツとして扱われていると推測される（Issue #160）。前提Issue #156（ガイドページの全ヘルプ再掲による重複解消）は対応済み・クローズ済みであり、ja側の構造的重複は解消済み。今回はen版の翻訳・コンテンツ自体の独自性向上が対象。
- **ゴール**: en版の各ページ（12ツールのヘルプ、トップ/menu/guideページ）を、ja版の逐語訳ではなく英語圏ユーザーの検索意図・表現に合わせた独自コンテンツとして作り込み、Googleのインデックス登録を回復する。
- **成功指標**:
  - Search Console上で en版の対象ページが「インデックス登録済み」に移行する（本対応の完了基準は「作り込み完了」までとし、インデックス登録自体はGoogleの再クロール後に確認するフォローアップ事項とする）
  - 全12ツールの en ヘルプ + トップ/menu/guideページが、後述の「独自コンテンツ要件」を満たした状態になる

## この方針にした理由
- Issue #160の背景記載のとおり、末尾スラッシュ不一致(#159)はcanonical処理で吸収されており直接原因ではないと確認済み。残る要因は「コンテンツの質」であり、特にen版で全ページに及ぶ深刻度から「機械的直訳」という構造的要因が濃厚と判断した。
- 一時的なnoindex化も検討されたが、Issue #160の対応方針として「作り込みを進める」ことが既に確定しているため、本書はその作り込みの具体的な方針・要件を定義する。
- ja版自体の文章修正は本方針のスコープ外とする（ja版はソース言語として現状維持し、構造的問題は#156で解消済み）。en版のみを対象に、翻訳プロセスを「直訳」から「ローカライズ＋独自加筆」に変える。
- 対応はフロントエンドエンジニアが`messages.en.xlf`の`<target>`を書き換える形で行うため実装コストは低い。新規コンポーネントや構造変更は不要であり、テンプレート（概要・使い方・仕様/用語解説・ユースケースの4セクション構成）自体は変更しない。テンプレート構造を変えると#156で解消した「量産コンテンツ」感が再発するリスクがあるため、変えるのは**本文の言葉選び・具体例・深さ**であり、**セクション構成や見出しラベルではない**。

## ユーザーストーリー
- 英語圏の開発者として、Googleで "sql formatter online" のように検索したとき、機械翻訳された説明ではなく、英語圏での実務的な悩み（例: ORM生成クエリの読みにくさ、レビュー時の規約統一など）に即した独自の解説を読みたい。それによって、このツールが「自分の困りごとに合っている」と判断し、安心して使い続けられる。

## フェーズ定義
| フェーズ | 対象機能 | 説明 |
|---|---|---|
| MVP（今回のスコープ） | 12ツールのen版ヘルプ4セクション + トップ/menu/guideページのen版コンテンツの独自化 | 既存テンプレート構造内で、本文を直訳から独自コンテンツへ書き換える |
| Phase 2（将来対応） | en版インデックス登録状況のモニタリング・追加チューニング | Search Console再クロール後の効果測定、必要であれば追加加筆。SEOキーワード調査ツールを使った精緻化も将来検討 |

## 機能要件

### 1. 翻訳プロセスの変更（直訳禁止ルール）
- `messages.en.xlf` の `<target>` は、対応する `<source>`（ja原文）の**逐語訳であってはならない**。
- 英語版は以下を許容・推奨する。
  - **構成順の入れ替え**: ja版で2文目に書かれている内容をen版では1文目に置く、など。
  - **具体例の差し替え・追加**: ja版にない英語圏特有の具体例（後述セクション3参照）を追加してよい。
  - **文の追加・削除**: en版独自に1〜2文程度の補足を追加してよい。ja版にある文をen版で省略するのも可（ただし4セクション構成・各セクションの主旨は維持する）。
  - 不可: Google翻訳・DeepLレベルの一文一文の機械的対応関係を保ったままの訳出。判定基準は「ja原文を見ずにen訳だけを読んで、英語ネイティブが書いた文章として違和感がないか」。

### 2. 各セクションの独自化要件
4セクション構成（概要・使い方・仕様/用語解説・ユースケース）は維持し、各セクションで以下を満たす。

- **概要 (Overview)**: ツールが解決する問題を英語圏の実務シーン基準で説明する。ja版の「何ができるか」の説明をそのまま訳すのではなく、「なぜ必要か（why)」を先頭に置く書き方を基本形とする。
- **使い方 (Usage)**: 操作手順は事実情報なので構造は概ね保ってよいが、UIラベルの英語表現は実際の画面の英語UIラベルと一致させる（直訳による不一致を避ける）。
- **仕様・用語解説 (Spec/Glossary)**: 英語圏で一般的な用語・略語があれば優先して使う（例: 後述の用語対応表）。ja語特有の言い回し（「〜を変更しない」等）はen版では英語の慣用表現に置き換える。
- **ユースケース (Use cases)**: en独自の落とし穴・Tips・実務ユースケースを最低1つ追加する（後述セクション3）。ja版のユースケースの直訳のみで終えない。

### 3. ツールごとの「実務的な落とし穴・Tips・英語圏特有表現」の組み込み方
en版ヘルプには、各ツールにつき**最低1項目**、以下のような英語圏読者向けの実務Tipsを追加する。フロントエンドエンジニアは実装時にツールごとに具体的な内容を調査・作成すること（本書では「何を追加すべきか」の枠組みのみ定義し、個別ツールの文章内容そのものは規定しない）。

追加すべき情報の種類（いずれか、または複数）:
- **よくあるエラー・誤用パターン**: 例: SQL整形ツールであれば「ORMが生成したSQLにコメントが混在しているとどうなるか」のような実務上ありがちな入力パターンへの言及。
- **関連する業界標準・仕様への言及**: 例: UUID生成ツールであれば RFC 4122 / RFC 9562 のような正式仕様名を明示し、英語圏の検索者が使う技術用語（"UUID v4 vs v7", "time-ordered UUID" 等）を本文に含める。
- **英語圏で使われる用語・略語の統一**: ja版の言い回しを直訳した結果不自然になる用語は、英語圏で実際に検索・使用される表現に置き換える。
- **比較・代替手段への言及**: 例: パスワード生成ツールであれば "password manager" との役割の違いに触れるなど、検索者が比較検討する文脈を補う。

これらはja版には存在しない・または簡略にしか触れられていない情報であってよい。ja版とen版で情報量が異なること（en版の方が厚くなること）を許容する。

### 4. トップページ・menuページ・guideページの独自性確保
ヘルプコンポーネントと同様に、以下のページのen訳についても直訳ではなく独自コンテンツとする。

- **トップページ（dashboard-page）**: サイト全体の紹介文。英語圏訪問者向けに「このサイトが何を解決するか」を独自の言葉で説明する。
- **menuページ**: 各ツールカードの`description`（`menu-def.ts`由来の文言）について、ja直訳ではなく検索意図に合わせた短い英語独自文に変更する。1ツールあたり1〜2文の短文のため、要約の質（キーワード選定）を重視する。
- **guideページ（#156対応済み）**: 各ツール紹介文（`page.guide.intro.*`）は既にリンク誘導用の短い導入文に変更済み。en版についても、ja版の直訳ではなく、各ツールページのen版ヘルプの概要セクションと**文言を変える**（完全一致させない）独自の要約文とする。ガイドページとツールページのen版本文が一致すると、#156で解消したはずの重複コンテンツ問題がen版で再発するため、この点は必須要件とする。

### 5. 対象ファイル一覧（実装時の参照用）
- ヘルプコンポーネント（12ツール、`*-help.component.html` 内の `i18n="@@..."` に対応する `messages.en.xlf` の `<target>`）:
  - `api-key-gen-help`, `color-palette-help`, `ip-cidr-help`, `json-formatter-help`, `password-gen-help`, `sql-formatter-help`, `svg-viewer-help`, `text-diff-help`, `ulid-gen-help`, `unix-timestamp-help`, `url-encoder-help`, `uuid-gen-help`
- ページコンテンツ:
  - `src/app/pages/dashboard-page/`（トップページ）
  - `src/app/pages/menu-page/` + `src/resources/menu/def/menu-def.ts`（descriptionのen訳）
  - `src/app/pages/guide-page/guide-page.component.html`（`page.guide.intro.*`系のen訳）
- 翻訳格納先: `src/resources/texts/def/messages.en.xlf`（`<unit id="...">` ごとの `<target>`を書き換える）

## 受け入れ条件
- [ ] 12ツールすべてのen版ヘルプにおいて、4セクション（Overview/Usage/Spec/Use cases）の`<target>`が、対応する`<source>`の逐語訳になっていない（文構成・具体例のいずれかが異なる）
- [ ] 12ツールすべてのen版ヘルプにおいて、ユースケースセクションに、ja版に存在しない英語圏向けTips・落とし穴・用語が最低1項目追加されている
- [ ] トップページ・menuページ・guideページのen訳が、対応するja原文の逐語訳になっていない
- [ ] guideページのen版各ツール紹介文が、同ツールのen版ヘルプの概要セクションと文言が重複していない（要約はするが一致はしない）
- [ ] `yarn ng extract-i18n` 実行後、`messages.en.xlf` に `<target/>` の空欄（未翻訳）が残っていない（`grep -n '<target/>' src/resources/texts/def/messages.en.xlf` で確認）
- [ ] 修正後、en版ビルド（`ng build --configuration=en`）が成功する

## エッジケース・異常系
| ケース | 期待する挙動 |
|---|---|
| ツールによって追加できる実務Tipsのネタが少ない場合 | 比較・代替手段への言及など、セクション3の4種類のいずれかは必ず1つ満たす。完全に思いつかない場合はPdM/COOに相談し、簡易な調査（Web検索等）を許可する |
| en版での文章追加によりページが長くなりすぎる場合 | 4セクション構成・1セクションあたり目安200〜400語程度を上限とし、冗長にならないよう調整する。量よりも検索意図との一致・独自性を優先する |
| UIの英語ラベルと「使い方」セクションの記述が将来ずれた場合 | UI変更時は対応するヘルプの「使い方」セクションも合わせて更新する（既存のメンテナンス運用と同様） |

## 非機能要件
- **レスポンシブ対応**: 既存ページ構造を変更しないため追加対応不要
- **アクセシビリティ**: 既存のHTML構造・i18n仕組みを維持するため追加対応不要
- **パフォーマンス**: 文章量がツールあたり数百語程度増える可能性があるが、SSGビルド時に静的HTML化されるため実行時パフォーマンスへの影響はない

## スコープ外
- ja版本文の修正（ja版はソース言語として現状を維持する）
- 新しいツールページ・新しいヘルプコンポーネントの追加
- ヘルプの4セクション構成自体の変更（セクション数や見出しラベルの変更は行わない）
- Search ConsoleでのインデックスID登録状況の確認・モニタリング自体（作り込み完了後のフォローアップとしてPhase 2で扱う）
- noindexタグの追加（Issue #160で「作り込みを進める」方針が既に確定しているため、本書では扱わない）
- SEOキーワード調査ツールの導入（将来必要であれば別途検討）

## 優先度
| ページ/ツール | 優先度 | 理由 |
|---|---|---|
| guideページ・menuページ・トップページ | Must | Search Consoleで「ほぼ全ページ未インデックス」の中核として明示的に挙げられている（`en/`, `en/menu`, `en/guide`）。サイト内のハブページであり、修正効果が他ページへの内部リンク評価にも波及する |
| sql-formatter / json-formatter / url-encoder / uuid-gen-help | Should | サイトの主力ツールであり検索流入が見込みやすい（汎用的な開発ツールで検索ボリュームが相対的に大きいと推測される） |
| 残りの8ツール（api-key-gen, color-palette, ip-cidr, password-gen, svg-viewer, text-diff, ulid-gen, unix-timestamp-help） | Should | 同等の対応が必要だが、上記4ツールに対応パターンが確立した後に同じ要件を適用すれば効率的に展開できる |

着手順序の提案: **(1) guide / menu / トップページ → (2) 主力4ツール → (3) 残り8ツール**。(1)を最優先にするのは、Issue #160の記載でこの3ページが明示的に未インデックスの代表例として挙げられているため、構造的な修正パターン（guideページとツールページの文言重複回避を含む）を最初に確立する必要があるから。

## リスク・依存関係
- 本対応の効果（インデックス登録の回復）はGoogleの再クロール・再評価に依存し、対応完了後すぐに結果が出るとは限らない。COOおよびユーザーへは「作り込み完了」と「インデックス登録」を別の完了基準として説明する必要がある。
- フロントエンドエンジニアが英語ネイティブでない場合、不自然な英語表現のリスクがある。可能であれば英文レビュー（ネイティブチェックや専用ツール）を一次レビューに加えることを推奨する（本書のスコープ外だが実装時の注意点として記載）。
- guideページとツールページのen版本文を独立に書き換える際、将来どちらかだけ更新されて再度文言が一致してしまう「ドリフト」のリスクがある。レビュー時に両者を突き合わせるチェックをコードレビュー観点に加えることを推奨する。
